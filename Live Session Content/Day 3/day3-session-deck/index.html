Part 1 · 70 min

Anti-Slop &
reviewing AI code.

Calibrated review, not exhausting review. Catch slop with a system and a checklist.
Part 1 · the problem
Tidy makes review harder, not easier.
😌 What your eye expects
Rushed code looks rushed — bad names, dead formatting, obvious gaps. You've spent years reading those tells. They're how you triage what to read carefully.
🤖 What you actually get
AI drafts are clean, idiomatic, well-named, and the tests are green. Every tell is gone. Plausible-but-wrong code now reads exactly like correct code.
🎯
So we stop reviewing by vibe and start reviewing by system — a named set of failure modes and a checklist that transfers.
Part 1 · know your enemy
The four faces of slop.
🎭 Plausible-but-wrong
A helper that reads cleanly and is named well — and computes the wrong thing. Read what it does, not what it's called.
📋 Copy-paste drift
Lifted from a neighbour with one argument left pointing at the wrong thing. The shape is right; one field is wrong.
🪦 Dead branches
A branch or line that can never run. Harmless-looking, and a tell that nobody read it.
🫥 Tests that assert nothing
Runs the code, "passes", never checks the contract. Green, covering air. The most dangerous one.
🔎
All four are planted in your Day 2 share diff. Hold this list — you're about to go hunting.
The posture for the whole day
The AI's draft is untrusted by default.
And the tidier it looks, the more suspicion it earns — because tidy code is the code people stop reading carefully.

Part 2 · 90 min

Adversarial review:
multi-hat thinking.

Get judgments that aren't the author's. Independence is the operating principle.
Part 2 · why independence
One reviewer holding everything holds nothing well.
🥽 All-in-one
"Make it secure, DRY, performant, correct, well-tested." It does each shallowly, and silently resolves the tradeoffs between them. You never see the argument.
🎩 A panel of single hats
Each session has one job and one vantage. When two hats disagree, the tradeoff a single reviewer would have buried becomes visible and debatable.
💡
The discussion is the deliverable. You're not collecting findings — you're surfacing the decisions nobody was making out loud.
Part 2 · the panel
Three hats, three sessions.
Hat 1
🔒 Security
"How does an attacker abuse this?" Access control, injection, guessable tokens, secrets. Guilty until proven safe.
Hat 2
📐 Design / DRY
"Correct and maintainable in six months?" Copy-paste drift, dead code, duplication, contract clarity.
Hat 3
📦 Black-box
Sees only the contract, not the code. Probes what the API promises. Cannot anchor, cannot cheat.
🎩
Separate sessions, on purpose. Prompts are in participant/multi-hat/. Same diff, three independent vantages.
Part 2 · the hat that can't cheat
Black-box = implementation-blind.
Why blind matters
A tester that can read the code anchors to it, and will quietly weaken a test so the code passes. That's the author judging its own work — the exact thing we're escaping.
Given only the contract, it can only probe what the contract promises. It proves the revoke bug without ever opening revoke_share.
# All the black-box hat is allowed to see: CONTRACT — share a note • owner mints a public link to their note • anyone with the link can read it (GET /s/<token>) • after revoke, the link MUST 404 • a link must not be guessable • only the owner may mint or revoke # It writes tests to break each promise.
One hard line · non-negotiable
Never ship AI code with only AI-written tests.
Keep at least one human-authored assertion on anything that matters. Today you write the revoke assertion yourself — and watch it fail against the planted code.
🧪 Lab 2 · ~45 min · the flagship
Run the panel vs. one reviewer.
What you do
  • Run security, design/DRY, black-box as 3 separate sessions
  • Run the single all-in-one reviewer as the control
  • Fill the comparison table — what only independence surfaced
  • Write one human-authored assertion on the critical path
  • Commit it — watch it fail, then know why
What you're proving
The black-box hat proves the revoked link still resolves from the contract alone. The design hat finds its cause in the code. The single reviewer, juggling everything, resolved it silently and moved on.
📤
Deliverable: panel-vs-single evidence + one committed human-authored assertion on your running build.
Part 3 · 90 min

Secure-by-default
& OWASP skills.

Catch issues while the agent writes. Standardise the checks. Adopt tooling as a security decision.
Part 3 · the same trap, sharper teeth
Insecure code is idiomatic too.
💉 Injection
An f-string in a SQL query reads completely normal. A03 ships looking like clean code.
🚪 Broken access control
A missing ownership check is an absence — nothing on screen looks wrong. A01, invisible.
🔑 Guessable secrets
A token built from an id "works" in every test. Predictable, and tidy enough to trust.
⬅️
The fix is to shift the check left — into the moment of writing — so secure-by-default tooling matters more when the output looks clean, not less.
Part 3 · the agent applies secure patterns as it writes
Install an OWASP skill.
🛡️ Secure as it writes
The agent applies OWASP patterns during generation — parameterised SQL, ownership checks, unpredictable tokens — not after.
🎓 Explains every flag
It says why a pattern is risky in plain language. That doubles as real-time coaching for the whole team.
📚 Reputable + comprehensive
Today: a project-scoped stand-in. Real world: a vetted skill like agamm/claude-code-owasp, installed the same way.
🔧
The skill catches the planted vuln while you build — and tells you which OWASP category it is and how to fix it.
Part 3 · make it the team's, not yours
Project scope + a security subagent.
📌 Project-scoped install
Commit the skill into .claude/skills/. Every engineer on the repo inherits the identical checks. Security stops being personal preference.
🤖 Independent security subagent
A dedicated security-reviewer runs an OWASP pass on any meaningful diff — the mechanized cousin of Part 2's security hat.
# committed in the repo — inherited by the whole team .claude/skills/owasp-review/SKILL.md # secure-by-default, project scope .claude/agents/security-reviewer.md # independent OWASP pass .claude/hooks/secret_scan.sh # deterministic backstop
Part 3 · adoption is a security decision
The marketplace is attack surface.
✅ Reputable sources
A skill runs with the agent's privileges — your code, your shell, the network. Provenance matters.
👀 Read before install
Open the skill and read it. You'd review a dependency; review this.
📌 Pin versions
Pin what you adopted. An auto-updating skill is an unreviewed change.
🔍 Treat as a review
Adopting a plugin is a security review. Run it like one.
⚖️
We just told you to install an OWASP skill. These four rules are how you install anything — including that one.

Part 4 · 40 min

Permissions without
constant interruptions.

Flow without recklessness — and the same posture for every engineer.
Part 4 · the three buckets
Allow · Ask · Deny.
✅ Allow
Routine, reversible, read-mostly. No prompt.
  • read files
  • pytest · ruff
  • read-only git
🟡 Ask
Consequential but normal. Prompt every time.
  • git commit · push
  • pip install
  • gh
⛔ Deny
Never, no prompt. A hard floor.
  • rm -rf
  • curl … | sh
  • read .env / keys
# .claude/settings.json — the committed starting posture "allow": ["Read","Bash(pytest:*)","Bash(ruff:*)","Bash(git diff:*)"], "ask": ["Bash(git commit:*)","Bash(git push:*)","Bash(pip install:*)"], "deny": ["Read(./.env)","Bash(rm -rf:*)","Bash(curl:* | sh)"]
Part 4 · the flow lever
Auto mode: a classifier, not a human.
⚡ The upside
A background classifier replaces manual prompts — it decides allow vs ask for you. Far fewer interruptions, real flow on routine work.
⚠️ The trade
You've moved a human gate to a model gate. When it misjudges, it does so silently — no prompt is the whole point, and also the risk.
🧱
Team position: enable it only with a hard deny floor underneath as a non-negotiable backstop. Per-repo, and revisit once you've lived with the baseline.
🧪 Lab 4 · ~25 min · team-wide
Set the posture, then argue one rule.
What you do
  • Start from the committed baseline in settings.json
  • Sort the agent's actions into allow / ask / deny
  • Decide your auto-mode position (+ the deny floor)
  • Debate one contested rule to a position
  • Commit it — dissent recorded in the message
The flashpoint
Should git commit be allow or ask? Flow says allow — the diff was already reviewed and the secret-scan hook gates it. Safety says ask — a human should see what ships. Argue it to a decision.
📤
Deliverable: a committed permissions baseline with the contested rule's reasoning recorded.
