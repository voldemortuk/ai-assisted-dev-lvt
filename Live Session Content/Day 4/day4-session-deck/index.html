Part 1 · 60 min

Skills:
teach the team once.

Capture what works into a skill that triggers reliably and the team inherits.
Part 1 · what good looks like
A skill that triggers and earns its keep.
🎯 Triggers reliably
The description names the real situations it applies to. Fires when it should, stays quiet when it shouldn't.
📂 Progressive disclosure
Loads only when relevant, so it can carry detail without taxing every session.
✂️ Earns its context
Every line changes behaviour. No essays, no restating the README.
📌
The worked example is committed: .claude/skills/repo-conventions/. Copy its shape for your own.
Part 1 · draft, test, trim, retire
A skill has a lifecycle.
1
✏️
Draft
Frontmatter + tight, behaviour-changing instructions.
→
2
🎯
Test triggers
Fires on the right task, quiet on the wrong one.
→
3
✂️
Trim
Cut anything that doesn't change a diff.
→
4
🗑️
Retire
Pull it when it goes stale.
⚠️
Testing the trigger is the step people skip. A skill that doesn't fire reliably is worse than none — it's clutter pretending to help.
Part 1 · adopt and distribute
Adoption is a security decision.
🛒 Adopting from the marketplace
High-value skills exist, but each runs with the agent's privileges. Prefer reputable sources, read before installing, pin versions, treat adoption as its own review. (Day 3's rule.)
📦 Distributing your own
Commit the skill to .claude/skills/ and the whole team inherits it. Keep a polyrepo coherent by committing the skill, not copy-pasting prose.
🎓
This is the same instinct as the Day 1 Prompt Pattern Library, now made first-class, committed, and triggering on its own.
🧪 Lab 1 · ~35 min · one skill each
Write a skill, prove it triggers.
What you do
  • Pick a candidate: Day 1 pattern, Day 2 spec, or Day 3 checklist
  • Draft SKILL.md, modelled on repo-conventions
  • Test the trigger: fires when it should, quiet when it shouldn't
  • Trim, then commit to .claude/skills/
The bar
A sharp description (it lives or dies here), instructions that change a diff, and proof of triggering in both directions. Guide in SKILL_LAB.md.
📤
Deliverable: one committed skill, proven to trigger on the right tasks.
Part 2 · 55 min

Hooks:
deterministic guardrails.

Guarantee critical behaviour on a lifecycle event, instead of hoping the model chooses to.
Part 2 · before and after
PreToolUse stops, PostToolUse follows up.
🛑 PreToolUse
Runs before an action. A blocking exit code 2 stops the agent. This is your guard: validate or refuse before anything happens.
🧹 PostToolUse
Runs after an action: format, lint, test, log. The "make sure it always happens" half. Advisory, exit 0.
🔒
Committed to settings.json, the whole team inherits the same gates. Exit codes are the contract.
Part 2 · the two you'll wire
Format, and guard.
🧼 Auto-format (PostToolUse)
format_on_edit.sh formats every Python file the agent touches. House style guaranteed, not hoped for.
⛔ Block dangerous (PreToolUse)
block_dangerous.sh refuses rm -rf, curl | sh, force pushes, with a blocking exit code.
# the guard fires before the command runs: $ echo '{"command":"rm -rf /"}' | bash .claude/hooks/block_dangerous.sh BLOCKED — refusing to run: rm -rf / exit 2 # the agent is stopped before anything happens
Part 2 · what belongs where
Hook it, or judge it.
⚙️ Belongs in a hook
Must always happen, checkable mechanically: format, secret scan, block rm -rf, run the tests.
🧠 Belongs to judgment
Needs reasoning: is this design right, is this fix correct, should we ship. Don't encode this in a hook.
🔗
These hooks are the enforcement layer behind Day 3's self-verifying loop and secret scan, and Day 1's responsible-AI checklist. "Hope it runs the tests" becomes "the tests run."
🧪 Lab 2 · ~25 min · prove they fire
Two gates the team inherits.
What you do
  • Read both hook scripts + the wiring in settings.json
  • Prove the guard: pipe rm -rf / in, see the block + exit 2
  • Prove the formatter: a bad edit gets tidied
  • Commit the settings; the team inherits both
What you're proving
A critical behaviour is now guaranteed, not left to the model. Guide in HOOKS_LAB.md. Hooks from scratch is a deeper topic; today you wire and prove the templates.
📤
Deliverable: two working hooks the team inherits, both proven to fire.
Part 3 · 70 min

Orchestration:
who holds the plan?

Four ways to run multi-step work. Subagents hands-on, the heavyweights demoed.
Part 3 · the escalation ladder
Four ways to run multi-step work.
💬 Interactive
You decide, results in your context. The default.
🤖 Subagent
Own context, scoped tools, summary back. The workhorse.
👥 Agent team
Lead + peers, shared list + mailbox. Heavy. Experimental.
🌐 Dynamic workflow
A script runs parallel subagents; only the final answer returns. Very heavy. Preview.
🪜
Climb the ladder only when the job justifies the cost: interactive → subagent → team → workflow.
Part 3 · why subagents win daily
Work happens elsewhere.
🪟 Own context
The research happens in the subagent's window. Your main session stays clean and on-task.
🔧 Scoped tools
It can only do what you allow. The provided researcher is read-only.
📋 Structured report
The conclusion and citations come back, not the whole search. Pick a cheaper model per subagent.
🤖
Agent teams are the mechanized form of Day 3's multi-hat panel. Today you run the workhorse and watch the heavyweights.
🧪 Lab 3 · ~40 min · hands-on + demos
Delegate, then watch the heavyweights.
What you do
  • Delegate a research task to the researcher subagent
  • Scoped tools + a structured report-back, main context clean
  • Watch the agent-team multi-hat demo (token cost on screen)
  • Watch the dynamic-workflow demo (run once, very heavy)
Honest framing
Nothing in the standard workflow depends on the experimental primitives yet. They're demoed so you know they exist and what they cost, not adopted. Guide in SUBAGENT_LAB.md.
📤
Deliverable: a subagent delegation pattern with scoped tools and a structured report-back.
Part 4 · 55 min

Tokenomics & context
economics in production.

What it costs at team scale, and the levers that control it.
Part 4 · where the bill comes from
Spend concentrates in a few places.
📄 File reads
Re-reading whole files instead of targeting interfaces.
🔧 Tool output
Noisy logs and test dumps sitting in context.
🔁 Retries
Re-rolls from a vague prompt or poisoned context. The silent killer.
💬 History
The whole conversation, paid for every turn.
🎯
Optimise cost-per-outcome, not cost-per-task. The cheapest session is the one that ships on the first pass — which is what Days 1 to 3 bought you.
Part 4 · the right model for the job
Route, but gate it.
🧭 The routing rule
  • Frontier for planning & review
  • Fast / cheap for mechanical edits & subagents
  • A quality gate on every routed path
⚠️ The false economy
Routing everything to the cheap tier to save money, then eating it back in re-rolls and a missed review finding. Routing without a gate is not a saving.
🛡️
The gate is what makes routing a saving instead of a silent regression. If a tier fails the gate, route it back up.
Part 4 · retrieve structure, not files
The AST context graph.
The idea
Index the repo into a graph of functions, imports, and call edges (a parser such as Tree-sitter). The agent retrieves the minimum relevant subgraph instead of re-reading files.
Reports cluster around ~70% token reduction (up to ~90% on big monorepos), with quality regressions when retrieval misses structure. So: pilot, measure both, decide.
# the repo ships a working version: $ python tools/code_graph.py cost get_note answering 'what depends on get_note?' subgraph retrieval : ~ 83 tokens whole-file reads : ~ 1083 tokens reduction : ~92% # pilot, then measure quality too
🧪 Lab 4 · ~30 min · build the model
Make the bill an engineering problem.
What you do
  • Instrument a task; name its two biggest cost drivers
  • Run one mechanical task on two tiers; compare cost vs quality
  • Write your routing rule (ROUTING_RULES.md)
  • Run code_graph.py cost; compare the token counts
  • Fill the cost model + context-graph pilot criteria
What you leave with
A cost model and decision framework for production adoption, with pilot criteria for context-graph tooling (a cost and a quality threshold). Worksheets in COST_MODEL.md.
📤
Deliverable: a measured session, a routing rule, and context-graph pilot criteria.
