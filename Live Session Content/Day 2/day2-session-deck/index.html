Part 1 · 55 min

Sizing &
scoping work.

Match the approach to the task. Stop reaching for the same move every time.
Part 1 · the menu
Four approaches, least ceremony first.
⚡ Quick
Interactive, no plan. Low blast radius, obvious done, glance-reviewable. A typo, a version bump.
🗺️ Plan mode
Drive through a plan, approve before code. Real blast radius or ambiguity. A rename, pagination.
📋 Spec-driven
Write the contract first. Multi-step, becomes a running artifact. The share feature.
🚫 No agent
Do it yourself. Novel architecture or a judgment call. "Should we move to Postgres?"
🎯
Not "always plan" or "always spec". Stop at the cheapest approach that's still safe.
Part 1 · how to score it
Three signals pick the approach.
💥 Blast radius
If this is wrong, how much breaks? One error string, or authentication for every user? High radius pushes you toward plan or spec.
🌫️ Ambiguity
Is "done" obvious, or does it need a contract written before anyone builds? High ambiguity means write the spec first.
🔍 Reviewability
Can you verify the result in seconds, or only by reading every line? Low reviewability earns a plan you can check up front.
🧭
Getting it wrong has a cost both ways: too small an approach drifts, too big a one wastes the room.
Part 1 · when it's too big
Break it at clean seams.
🧩 Agent-sized chunks
A large task becomes 4 to 6 pieces, each small enough to drive in one pass and independently reviewable.
🔗 The seam is the point
A stable interface between chunks so a mistake in one doesn't leak into the next. Define the seam before you build either side.
✂️
Bad chunking just moves the mess around. Good chunking puts a reviewable contract at every seam.
🧪 Lab 1 · ~25 min · pairs
Classify the backlog, chunk the big one.
What you do
  • Open BACKLOG_12.md — classify all 12 by approach
  • Write a one-line reason per item; defend it to your pair
  • Take the oversized OAuth ticket (item 11)
  • Break it into 4 to 6 chunks with explicit interfaces
The lens
For each item, score blast radius, ambiguity, and reviewability, then read the approach off the rubric. The chunking seam to find: keep current_user() stable so every route is insulated from the swap.
📤
Deliverable: a defended 12-item classification + a chunked plan with clean seams.
Part 2 · 75 min

Plan mode: the cheapest
place to catch a mistake.

Challenge the plan before a single line is written.
Part 2 · the economics
Review the plan, not the diff.
📝 The plan is cheap text
The agent states its steps, files, and order before writing code. You review intent, not implementation. Correcting a plan is one line.
💸 The diff is expensive code
The same mistake found later costs more at every stage: a re-run in build, a round trip in review, an incident in production.
📉
Plan mode moves the catch upstream, where mistakes are cheapest. That's the whole reason it pays for itself.
Part 2 · read it like a reviewer
Four ways a plan lies to you.
🕳️ Missing steps
The plan skips what the task actually needs. The headline requirement quietly dropped.
📁 Wrong files
A step edits the wrong module. Reads fine, points at the wrong place.
🎭 Hidden assumptions
Assumes an empty table, fresh data, a happy path. Breaks on reality.
📈 Scope creep
Gold-plates past what you asked. Extra surface, extra risk, unrequested.
🔢
The tidy numbering is the trap. Read what each step does, not how neat the list looks.
Part 2 · once you find the flaw
Intervene, then approve.
🛠️ Four moves
  • Edit a step directly
  • Constrain the approach
  • Re-plan from a correction
  • Abandon it if the premise is wrong
🤝 A shareable artifact
The plan is reviewable text. Hand it to a teammate before implementation cost is sunk and get a critique in minutes, not after a day of code.
✅
The rule: never approve a plan you haven't challenged at least once.
🧪 Lab 2 · ~40 min · pairs
Drive a plan, then break one.
What you do
  • Drive a non-trivial backlog change entirely through plan mode
  • Intervene on the plan at least once before approving
  • Open FLAWED_PLAN.md — find the planted defects
  • Re-plan it correctly; pairs swap and critique
The flawed-plan drill
The provided plan looks reasonable and is wrong in at least three ways: a step on the wrong file, a missing backfill (the actual task), an unstated assumption about existing data, and a verification that verifies nothing.
📤
Deliverable: a change shipped through plan mode with a recorded intervention + the flawed plan corrected.
Part 3 · 90 min

Spec-driven
development.

Turn a PRD into a spec the agent can be tested against. Build the running feature.
Part 3 · right-size the spec too
Lightweight or full.
🪶 Lightweight spec
A handful of acceptance criteria plus constraints. Right for a small, well-understood change. Fast to write, enough to test against.
📚 Full spec
Adds interfaces, non-goals, and a detailed definition of done. Right when blast radius is high or the contract is contested.
🎯
The same sizing instinct as Part 1: write the smallest spec that makes the agent testable.
Part 3 · what makes it executable
Four parts the agent can be tested against.
✅ Acceptance criteria
Each one a contract: given X, when Y, then Z. If you can't write it as a test, it isn't a criterion yet.
🚧 Constraints
What to reuse, what not to do, the security and style rules the agent must follow.
🛑 Non-goals
Written down, so the agent doesn't gold-plate. Closes the door on scope creep.
🏁 Definition of done
The gate: all tests pass, lint green, plus one human-authored assertion on the critical path.
🔗
Point the agent at the PRD and tech spec you already have, then add this thin executable layer on top. Don't rewrite what exists.

Part 4 · 65 min

CLAUDE.md: project memory
that earns its keep.

Write it, measure the behaviour change, then trim the fat.
Part 4 · what earns a line
Belongs, and doesn't.
✅ Belongs
  • build / test / run commands
  • conventions the agent keeps missing
  • an architecture pointer
  • rules you re-type into prompts
⛔ Doesn't
  • prose essays and philosophy
  • anything the repo already says
  • stale rules (worst: they mislead)
  • decoration that changes nothing
📏
The test for any line: would the agent behave differently without it? If not, cut it.
Part 4 · every line has a cost
Trimming is the discipline.
💰 Paid every turn
Day 1's lesson applied: each line costs context on every turn. A 15-line file that changes behaviour beats a 60-line one that decorates.
🗂️ Multi-directory
For larger repos and monorepos, a CLAUDE.md per directory keeps each area carrying only its own rules. Brief, but worth knowing.
📉
Adding is easy and feels productive. The skill is cutting the lines that don't earn their keep. Day 4 prices this in dollars.
🧪 Lab 4 · ~30 min · before / after / trim
Measure it, then cut a third.
What you do
  • Run the GET /notes task with no CLAUDE.md; save before.diff
  • Write a CLAUDE.md: commands, conventions, one pointer
  • Re-run the same task; save after.diff
  • Diff the two — that's the behaviour change
  • Trim ≥33% in a team review pass
What "after" should look like
Parameterised query, reuses the ownership accessor, route and test land in house style on the first pass, fewer questions asked. Calibrate the trim against CLAUDE_MD_BLOATED_EXAMPLE.md.
📤
Deliverable: a committed CLAUDE.md with before/after evidence, trimmed by a third.
