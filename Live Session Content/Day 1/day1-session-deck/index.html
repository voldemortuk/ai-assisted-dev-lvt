Part 1 · 60 min

How coding agents
actually work.

Build the model first, so you can predict failure instead of being surprised by it.
Part 1 · under the hood
Prediction, not retrieval.
🎲 Next-token prediction
The model predicts the next token over its context, sampling from a distribution. It isn't looking up an answer, it's generating a likely one.
🔀 Same prompt, two diffs
Sampling means the same prompt can produce different output twice. That's by design, not a bug. We'll run it live and watch.
🧪
This is why you review every diff and why reproducibility comes from specs and tests, not from hoping for the same output.
Part 1 · the agent's memory
The context window is everything it knows.
📜 System prompt
The agent's standing instructions.
🧠 Project memory

📄 File reads + tool output

💬 Conversation so far

It's finite. What fits, it uses. What falls out, it forgets. Managing that window is the highest-leverage habit there is.

Part 1 · a frank map
Where it's strong, where it's weak.
💪 Strong
  • Boilerplate & scaffolding
  • Test generation
  • Migrations & mechanical refactors
  • Comprehending unfamiliar code
🚧 Weak
  • Novel architecture
  • Ambiguous requirements
  • Long-horizon consistency
  • Subtle concurrency
🎯
The skill: know which column a task lands in before you start. Sometimes the right call is to leave the tool off.

Make the model concrete.
What you do

  • Will it fail? predict success/struggle/fail on 10 tasks
  • Token budget: itemise your context, rank what to evict
What you're proving
Non-determinism is real (so review every diff), failures are structural and predictable (so you can plan around them), and context is finite (so curation matters). Cards in WILL_IT_FAIL_CARDS.md.

Part 2 · what to check, in order
Read what runs.
1 · Behaviour first
Run it in your head on the tricky input. What does it actually do?
2 · What tidiness hides
Clean code earns more scrutiny, not less. Don't be lulled.
3 · Does the test assert?
The contract, or just that something returned? Green can cover air.
4 · When in doubt, reject
A reject costs a re-prompt. A bad accept costs an incident.
🎓
These reps tune your eye to AI failure modes before deeper trust gets extended on Day 3.
Part 2 · what the prompts protect
Today runs in approval mode.
🔐 Permission prompts
Claude Code asks before consequential actions (running a command, writing a file). The prompt tells you what it's about to touch. Read it.
📅 The full strategy is Day 3
Today: approve actions as they come. The posture of what to allow, ask, and deny is a whole block on Day 3. For now, don't reflex-approve.
🧼
Session hygiene from hour one: start clean, one task per session, check what the agent is actually seeing.
🧪 Lab 2 · ~45 min · the first ship
Your first committed change.
What you do
  • Pick one small change from BACKLOG.md
  • Drive it end to end: prompt → review → test → commit
  • Run the diff-review drill: 3 AI diffs, rising subtlety
  • Call accept / reject / amend on each; defend it
The three diffs
Diff 1 hides the bug behind a bare try/except. Diff 2 returns everyone's notes (missing the owner filter). Diff 3 is tidy, has a test, and is still wrong (validates after the write, test asserts nothing). Files in diff-review-drill/.
📤
Deliverable: one committed, tested change + three defended review calls.
Part 3 · 70 min

Context engineering: the skill
behind every good session.

Decide what enters the window, what stays out, and when to reset.
Part 3 · what's in the window
Everything competes for space.
🧠 Project memory
CLAUDE.md and standing rules.
📄 File reads
Every file pulled in, still sitting there.
🔧 Tool output
Logs, test runs, command output.
💬 History
The whole conversation so far.
📉
Crowding degrades output: worse edits, drift off task, slower sessions, higher spend. Clutter is a cost you pay every turn.
Part 3 · curate on purpose
Scope the working set.
🎯 Reference what's needed
Point at the exact files the task touches, and at interfaces rather than full implementations. The agent gets the contract without the noise.
🧹 Keep exploration out
Once you've found what you need, the exploration reads are dead weight. A build session carries only what the build requires.
🧭
A tight working set is the difference between a focused agent and a drifting one.
Part 3 · know when to reset
Clear, compact, or start fresh.
🔄 Clear vs compact
Clear when the task changes. Compact when the thread is long but still relevant. Different tools for different problems.
☠️ Signs of a poisoned context
It repeats itself, re-asks what you told it, edits the wrong file, drifts. When you see them, a fresh session usually wins.
👥
This compounds across a team once you share a vocabulary for naming context problems during review. Day 4 prices it in tokens.
🧪 Lab 3 · ~30 min · the centerpiece
Make a failing task succeed.
What you do
  • Run a task on a deliberately bloated context; watch it drift
  • Re-run it in a fresh, curated session; compare the diffs
  • Audit what your session is carrying; rank by cost
  • Write your personal eviction rule
  • Swap a failing transcript; diagnose your pair's
What you're proving
The same task, same model, same change-prompt, succeeds or fails on what else was in the window. You watched it fail and then recovered it. Worksheet in CONTEXT_AUDIT.md.
📤
Deliverable: a side-by-side bloated-vs-curated comparison + a context checklist you'll reuse.
Part 4 · 60 min

Prompting for developers
& codifying house patterns.

Write prompts with the discipline of a good ticket. Pin output to house style.
Part 4 · write it like a ticket
Four parts of a strong prompt.
🎯 Specificity
The exact file, endpoint, function, behaviour. Not "the notes endpoint".
🚧 Constraints
What to reuse, what not to change, the rules to follow.
✅ Success criteria
The behaviour, stated as something checkable. "Missing body returns 400."
🏁 Definition of done
The gate: tests pass, the case is covered, lint clean.
🧪
The test: could an engineer who's never seen this repo produce the right change from your prompt alone? If not, it's underspecified.
Part 4 · steer harder, avoid traps
Worked examples + anti-patterns.
📌 Worked examples
Show one canonical snippet from the repo (say get_note_for_owner) and output matches conventions on the first pass. It steers harder than three sentences of description.
🚫 Anti-patterns
  • Vague asks ("make it better")
  • Over-asking in one prompt
  • Missing acceptance criteria
  • Prompting for code when it's a design question
📚
Capture the prompts that work into a shared library. Personal trick becomes team asset — the seed of Day 4's skills.
🧪 Lab 4 · ~30 min · seed the library
Weak prompt to strong.
What you do
  • Rewrite 3 weak prompts (WEAK_PROMPTS.md) to ticket standard
  • Run before and after; compare the diffs
  • Pin house style with a worked example from the repo
  • Contribute one tested prompt to the Pattern Library v0
What you're proving
A strong prompt produces reviewable, convention-matching code in one or two passes, not five. The team leaves Day 1 with the start of a shared library that grows all week.
📤
Deliverable: before/after rewrites with diffs + one entry in PROMPT_PATTERN_LIBRARY_v0.md.
Part 5 · 30 min

Responsible AI for
AI-assisted engineering.

Protect LVT's code, data, and customers from day one.
Part 5 · five concerns
Protect code, data, and customers.
🔒 Data boundaries
No secrets, credentials, or customer PII in a prompt or the context. Ever.
💉 Injection
Instructions hide in files, deps, web content. Don't let the agent run what it found unread.
🌀 Hallucination
Invented APIs, plausible-but-wrong code. Tidy output still needs verifying.
✍️ Accountability
Whoever ships owns the change, regardless of who wrote it.
🛡️
Also: provenance & licensing of generated code. The permission model (Day 3) is the backstop for injection; this checklist is the first line.
🧪 Lab 5 · ~15 min · flag & fix
Spot the risky ones.
What you do
  • Flag 5 prompts/transcripts: exposure, injection, both, or fine
  • Rewrite one flagged prompt to be safe and still useful
  • Keep the responsible-AI checklist as a team artifact
The set
Two paste live credentials or PII. Two ask the agent to run instructions it found in a file or on the web. One is fine. Files in responsible-ai/.
📤
Deliverable: flagged risks + one safe rewrite + the checklist (enforced with hooks on Day 4).
